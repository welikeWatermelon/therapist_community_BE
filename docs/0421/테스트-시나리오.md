## Step 4: search_text 백필 확인용 SQL 추가

시드 데이터가 SQL INSERT로 들어가면 `search_text`가 NULL입니다.
V25 마이그레이션이 `WHERE search_text IS NULL`인 행을 자동 백필하지만,
**data-local.sql이 Flyway 이후에 실행**되므로 별도 백필이 필요합니다.

`data-local.sql` 맨 끝에 아래 추가:

```sql
-- === search_text 백필 (시드 데이터용) ===
UPDATE therapy_posts
SET search_text = TRIM(
    COALESCE(title, '') || ' ' ||
    COALESCE(LEFT(content, 100), '') || ' ' ||
    CASE therapy_area
        WHEN 'UNSPECIFIED'         THEN '선택안함'
        WHEN 'SENSORY_INTEGRATION' THEN '감각통합'
        WHEN 'SPEECH'              THEN '언어치료'
        WHEN 'OCCUPATIONAL'        THEN '작업치료'
        WHEN 'COGNITIVE'           THEN '인지치료'
        WHEN 'PHYSICAL'            THEN '물리치료'
        WHEN 'ART'                 THEN '미술치료'
        WHEN 'MUSIC'               THEN '음악치료'
        WHEN 'PLAY'                THEN '놀이치료'
        WHEN 'BEHAVIOR'            THEN '행동치료'
        ELSE ''
    END || ' ' ||
    CASE age_group
        WHEN 'AGE_0_2'     THEN '0세 2세'
        WHEN 'AGE_3_5'     THEN '3세 5세'
        WHEN 'AGE_6_12'    THEN '6세 12세'
        WHEN 'AGE_13_18'   THEN '13세 18세'
        WHEN 'AGE_19_64'   THEN '19세 64세'
        WHEN 'AGE_65_PLUS' THEN '65세 이상'
        ELSE ''
    END
)
WHERE search_text IS NULL;

-- === popularity_score 백필 ===
UPDATE therapy_posts SET popularity_score =
    (SELECT COUNT(*) FROM therapy_post_reactions WHERE post_id = therapy_posts.id) * 30
  + (SELECT COUNT(*) FROM therapy_post_scraps WHERE post_id = therapy_posts.id) * 20
  + CAST(EXTRACT(EPOCH FROM created_at) / 8640 AS BIGINT)
WHERE popularity_score IS NULL OR popularity_score = 0;
```

  -H "Authorization: Bearer $TOKEN"

# 빈 문자열 → 400

  "http://localhost:8080/api/v1/posts/search?keyword=" \
  -H "Authorization: Bearer $TOKEN"
# 기대: 400

# 공백만 → 400
curl -s -o /dev/null -w "%{http_code}" \
  "http://localhost:8080/api/v1/posts/search?keyword=%20%20%20" \
  -H "Authorization: Bearer $TOKEN"
# 기대: 400

# 101자 → 400
LONG_KW=$(python3 -c "print('가'*101)")

  "http://localhost:8080/api/v1/posts/search?keyword=$LONG_KW" \
  -H "Authorization: Bearer $TOKEN"
# 기대: 400

# size 51 → 400
curl -s -o /dev/null -w "%{http_code}" \
  "http://localhost:8080/api/v1/posts/search?keyword=감각통합&size=51" \
  -H "Authorization: Bearer $TOKEN"
# 기대: 400

## Step 9: DB에서 직접 검증

```bash
# psql 접속
docker exec -it builders-db psql -U builders -d builders
```

```sql
-- search_text 백필 확인
SELECT id, LEFT(search_text, 50) FROM therapy_posts LIMIT 5;

-- 인덱스 확인 (title_trgm, content_trgm 없어야 함)
SELECT indexname FROM pg_indexes WHERE tablename = 'therapy_posts';
-- 기대: idx_therapy_posts_search_text_trgm, idx_therapy_posts_created_at_id, PK만

-- title_choseong 컬럼 없어야 함
\d therapy_posts
-- title_choseong 없으면 OK

-- word_similarity 직접 테스트
SET pg_trgm.word_similarity_threshold = 0.1;
SELECT id, word_similarity('감각통합', search_text) AS score
FROM therapy_posts
WHERE '감각통합' <% search_text
ORDER BY score DESC;

-- EXPLAIN으로 인덱스 활용 확인
EXPLAIN ANALYZE
SELECT id, word_similarity('감각통합', search_text) AS score
FROM therapy_posts
WHERE '감각통합' <% search_text
ORDER BY score DESC LIMIT 10;
-- 기대: Bitmap Index Scan on idx_therapy_posts_search_text_trgm

-- bare LIKE 인덱스 활용 확인
EXPLAIN ANALYZE
SELECT * FROM therapy_posts
WHERE search_text LIKE '%감각통합%';
-- 기대: Bitmap Index Scan (데이터 적으면 Seq Scan일 수 있음 — 30행이라 정상)
```

### 30행에서 EXPLAIN이 Seq Scan 나올 때
정상입니다. PostgreSQL은 데이터가 적으면 인덱스보다 Seq Scan이 더 빠르다고 판단합니다.
100행 이상에서 Bitmap Index Scan이 나오면 정상.
